Background

The Design Guide is a means of unifying all design choices on the PEACH project. The website, for example, will specify design points such as typeface styles and sizes, within a dedicated design tab. It will also include a list of reusable components (and the accompanying code) that may be used within PEACH development projects.

Initially, this list will comprise solely components pulled directly from the development framework ( react-bootstrap), however will over time be updated by components built specifically for the PEACH project. Given the heavy use of data visualisation within PEACH, the site will contain a similar list for charts built using D3. Here users can see examples, alongside the code used to generate them.


Problem Definition:
  • Developers use components or a framework incompatible with the rest of the PEACH project, leading to inability to integrate their work, therefore, inability to divide the whole project into sub-tasks which could be assigned to different teams.

  • PEACH websites do not have a uniform look, leading to a more confusing user experience and less intuitive user interface requiring long learning process.

  • Developers are unsure of how to tackle the issue of data visualisation, for example what charts to use, and how to generate charts using React D3 (given how little documentation is in place).

    Gathering & Methods

    Elicitation
    The main method we employed was structured interview with the client. Due to the strict in-house nature of our project, our client wanted the design guide to behave a certain way. Additionally, we also conducted structured interviews with developers about what they would like to see in a design guide.
    The reason that we predominantly used interviews for requirement elicitation is due to the limited scope of participants within this project. While large scale questionnaire would certainly bring beneficial statistical analysis to commercial products aimed at the masses, our project is more specialised and aims to cater the needs of a small and dedicated group of peach developers. Thus we felt the best way for us to elicit requirements was to ask our client on prepared questions ranging from topics such as UI and content. These conversations would be recorded via Google Docs or otherwise and later refined into our requirement specification.

    Lab-sessions & Meetings
    We had discussions usually in the range of half an hour with the client almost every lab-session. During weekly meetings we presented our work for the week and asked for what our client wanted to be improved upon. Often, this was in the form of Bi-weekly sprints as it usually took us two weeks to implement a new feature or section of content. We then would proceed to iterate through our design process and implement the newly requested features. We have also arranged for private meetings with Navin, usually outside of UCL, where we could furthermore discuss about the requirements of the project when the time given within lab sessions was insufficient.

    Interviews
    Prior to the interviews, we prepared questions, research and any sketches or implementation we had done to show to our client. We understood the common belief that clients “do not know what they want until they see it”. Thus we opted to use visual techniques such as sketches and demos to demonstrate our ideas. The ultimate goal was to reach a common vision of the design guide between the client and us.
    During the interviews, one of us would be in charge of documenting any vital information, one would be actively engaged with the client while the last listened carefully for anything the others may have missed. We resolved any miscommunications between us and the client by using sketches and diagrams. It was important for us to fully understand our client’s perspective.

    Storyboard

    MoSCoW

    The Design Guide must follow professional medical User Interface guidelines:
    Using the pre-existing Common User Interface guidelines, created by the NHS, we tailored pre-existing libraries of components and design specifications to fit the PEACH project. Colour schemes were modified to fit the most commonly used colour codes employed by the medical community. Detailed information regarding form and data entry, based on the CUI guidelines, was also written up within the components page.
    The Design Guide must be of a professional standard:
    Throughout the components library, given that it is open source, many of the component examples and labels were unprofessional. These were all tidied up. Following this, the components library was edited to give a PEACH look and feel. Finally, components we deemed to be superfluous to the project needs, such as Custom Drop-Down and Custom Styles were removed from the components page. This was because the team decided these components offered too much freedom to deviate from prescribed design guidelines on the site, standing in direct contention of the purpose of a design guide.
    The Design Guide must contain code for PEACH components:
    Within the first iteration (pure HTML and CSS) of the Design Guide, the team was unable to display the code of the React components. Following the decision to rebuild in React, this issue was resolved. We were also able to implement a live code editor, allowing users to edit the example components and see the results in real time. This was well received by the client. Although we attempted to implement a similar system for the interactive D3.js examples, this was unsuccessful, we thus reverted to static code displays for this page.
    The Design Guide should contain embedded component examples:
    The Contents tab contains a list of interactive embedded components. At present, all components are default React Bootstrap components, however this will be updated when new custom PEACH components are developed.
    The Design Guide should contain design element guidelines:
    There currently exists no official guidance with regard to design for NHS websites. Furthermore, the client did not have a clear idea of what a PEACH site should look like. Upon the client’s suggestion, we therefore took a pre-existing design system (Lightning Design System) and ported the contents to the site, effectively acting as a placeholder. Again, this was edited to give a PEACH look and feel. Furthermore, sections we felt were not relevant to the project were removed.
    The Design Guide should be scalable and compatible with mobile devices:
    The site is fully scalable and compatible with mobile devices. Following the move from HTML and CSS, we were able to incorporate a React-Bootstrap framework to ensure the site is fully scalable and mobile compatible.
    We should provide documentation on how to add/edit site contents:
    The client wishes for to be able to edit certain portions of the site himself. Given that he is not familiar with React, and React sites tend to be more complex than traditional HTML/ CSS projects, we have written documentation describing which files are editable and how he can edit contents using only HTML and CSS.
    The Design Guide should contain sidebar integration:
    Each subpage within the Design Guide (D3, Design and Components) contains a "scrollspy" sidebar. This offers all the functionality of a traditional sidebar, whilst if the user chooses to scroll, the "scrollspy" keeps track of page location and marks it by highlighting the appropriate section on the sidebar.
    The Design Guide could include D3 examples:
    Later within the project, the client wished for us to display examples of D3 on the site. However, he was unsure of what form the examples should take. Accordingly, we decided to simply showcase some examples of interactive D3 charts that could be used in future PEACH projects, along with the accompanying code. This is in line with the Design Guide acting as the central reference point for all PEACH development, including not only design requirements and components, but also the necessary parts for data visualization.
