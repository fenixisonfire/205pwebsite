Achievements. Evaluation. Future.

Summary of Achievements

  • Fully functional React Bootstrap PEACH Design Guide site in place
  • Interactive components list, with embedded examples
  • Example components and descriptions modified to fit PEACH and NHS CUI guidelines
  • Live code editor accompanying each component
  • Embedded interactive D3 example displays in visualisation page
  • Static code display for each D3
  • Scrollspy within navbar at side of each page
  • Lightning Design System ported as placeholder for Design page content.

    • Critical Evaluation

    The team adapted well to a rapidly shifting set of requirements. What set out to be a somewhat insignificant part of the project quickly became the central focus. We have accommodated the client’s ever increasing ambitions for the site, and it has grown from a simple components list, to a multi faceted design guide, complete with notes on design philosophies. The components list now has interactive examples that can be edited by site users on the fly, using the live code editor. The visualisations tab, a relatively recent requirement to the project has been fully implemented, the examples adding to the overall aesthetic of the project.
    What We Did Well
    • The team was able to bring sophisticated functionalities to the Design Guide including live code editors, embedded D3 displays and interactive component examples.


    • The team’s initial priority requirement was not for the Design Guide, but a sub component for the form generator. As this project was scrapped halfway into the year, we were able to pivot to building a fully capable Design Guide.


    • Frequent use of drop down boxes means users do not experience information overload when using the site.


    • We were able to take loose ideas and requirements from the client, and use this freedom to create a Design Guide that exceeded his expectation both in aesthetic and functionality.


    • After realizing that using pure HTML and CSS to display react components was fundamentally flawed, we scrapped the first iteration of the design site. By quickly getting to grips with React Bootstrap, we were able to create a site with much better functionality.


    • User test responses to the site architecture has been overwhelmingly positive. The individual Components, D3 and Design tabs, coupled with logical subsections and the navbar, mean site navigation is highly intuitive.


    • As shown in the compatibility tests, the site functions well across all devices including tablet and mobile. The responsiveness of the site means function and aesthetics are not sacrificed when moving between screen resolutions.


    • The use of React in the site means the site is much more responsive. For example, React’s rendering by container system means there is much less unnecessary re-rendering of pages whenever the live code editor is used.
    What we can improve on.
    • As a team, we were should have pressed more for a concrete set of requirements. Far too often we succumbed to ‘scope creep’, where more and more requirements were added to the project.


    • We were unrealistic at the start of the project with regard to what could be delivered. We readily accepted a multitude of sub-projects that were either found to be redundant and cancelled, or found to be far too ambitious in scope to be achieved within our timeframe.


    • The site could be more efficient. Currently there are React components and sections in Design (such as voice and tone) that are unlikely to be used by Peach developers.


    • Although overall site architecture was implemented well, there are some large sub-sections within the Components page, such as ‘forms’ that could be further broken down to facilitate navigation.


    • Despite the site’s responsiveness, it is not possible to maintain the presence of the navbar in smaller screen sizes. Navigation within tabs becomes much more arduous therefore. As suggested previously, tab content could perhaps be further broken down into smaller sub pages.


    • At present, as revealed in user tests, the function of the D3 tab is unclear to some users. We should have perhaps either added more thorough information so that it could act as a tutorial page for complete data visualisation beginners, or make its purpose as a D3 showcase more clear.


    • Despite our success in rebuilding the site in React, it has not escaped our attention that many web developers are not proficient with this technology. This means site maintainability becomes much harder. To address this, we have produced a guide that describes how the site can be modified using only HTML and CSS.


    • User testing could have been more thorough. Many of the questions within the questionnaire were too vague. We should have asked more specific questions, regarding for example information layout. There should also have been more of an effort to engage with Peach development teams throughout development to see what features they would have liked included.
    Overall, the team has done well to create a fully functioning design site that has exceeded both our and the client’s initial expectations. Functionally and aesthetically, it is far beyond what was envisioned at the start of the project. Beyond this strong technical display, the team found itself wanting in terms of requirements engineering. We must accept partial blame for the flawed requirements, given that we were unrealistic as to what we could achieve at the start of the project.

    Future Work

    Future Development Plans

    The Design Guide in its current state is fully functional, and is able to serve its purpose, however there remain some features we wished to implement, but were unable to do so.

  • At present the site is not live, it will need to be deployed on Azure.
  • Adding live code editors to the D3 examples, so developers can modify examples instantly.


    • Collaboration opportunities

    The Design Guide is a continuously evolving project, dependent on many different PEACH teams to develop new content for the components list and the visualisations page. There may also be opportunities to look outward with regard to the site development.

    Throughout the project, we have heavily utilised the React-Bootstrap library, both in building the site and to populate the components list with default library components. As the number of components built specifically for PEACH increases and are added to the Design Guide list, as they will all be developed using React Bootstrap, there is the opportunity to either fork from the React-Bootstrap library and create a new PEACH React-Bootstrap library, or directly collaborate with the React Bootstrap developers, and contribute to their github, particularly as many PEACH components have use outside of healthcare analytics (such as the form generator).

    As mentioned previously, the Sales Force design system is a placeholder for design section content until the client finalises his own design schematics. However, there is no clear NHS design philosophy that the client must adhere to, and the Sales Force system can be used for a broad range of applications, therefore, it may be a better idea to collaborate with the Sales Force team and develop a slightly more restrictive, uniform version of their guidelines to fit with a medical site, and maintain uniformity of PEACH sites.

    We used the React D3 library for data visualisation. Although we found this library to be the most effective and easy way of embedding D3 charts within a React site, it is unfortunately no longer maintained. Given that the PEACH project is to generate a significant amount of content with regard to data visualisations, and that all PEACH sites will be React Bootstrap based, it may be beneficial for PEACH developers to take over the maintenance of this github site, and contribute to it.
